5.5.3 Compiling Combinations

The essence of the compilation process is the compilation of function applications. The code for a combination compiled with a given target and linkage has the form \begin{JavaScript} $\langle \textit{compilation of operator, target}$ proc, $\textit{linkage}$ next$\rangle$ $\langle \textit{evaluate operands and construct argument list in}$ argl$\rangle$ $\langle \textit{compilation of procedure call with given target and linkage}\rangle$ \end{JavaScript}

The registers {\lstinline[mathescape=true]$env$}, {\lstinline[mathescape=true]$proc$}, and {\lstinline[mathescape=true]$argl$} may have to be saved and restored during evaluation of the operator and operands. Note that this is the only place in the compiler where a target other than {\lstinline[mathescape=true]$val$} is specified.

The required code is generated by {\lstinline[mathescape=true]$compile_application$}. This recursively compiles the operator, to produce code that puts the function to be applied into {\lstinline[mathescape=true]$proc$}, and compiles the operands, to produce code that evaluates the individual operands of the application. The instruction sequences for the operands are combined (by {\lstinline[mathescape=true]$construct_arglist$}) with code that constructs the list of arguments in {\lstinline[mathescape=true]$argl$}, and the resulting argument-list code is combined with the function code and the code that performs the function call (produced by {\lstinline[mathescape=true]$compile_procedure_call$}). In appending the code sequences, the {\lstinline[mathescape=true]$env$} register must be preserved around the evaluation of the operator (since evaluating the operator might modify {\lstinline[mathescape=true]$env$}, which will be needed to evaluate the operands), and the {\lstinline[mathescape=true]$proc$} register must be preserved around the construction of the argument list (since evaluating the operands might modify {\lstinline[mathescape=true]$proc$}, which will be needed for the actual function application). {\lstinline[mathescape=true]$Continue$} must also be preserved throughout, since it is needed for the linkage in the function call. \begin{JavaScript} function compile_application(exp, target, linkage) { const proc_code = compile(operator(exp), "proc", "next"); const operand_codes = map(operand => compile(operand, "val", "next"), operands(exp)); return preserving(list("env", "continue"), preserving(list("proc", "continue"), construct_arglist(operand_codes), compile_procedure_call(target, linkage))); } \end{JavaScript}

The code to construct the argument list will evaluate each operand into {\lstinline[mathescape=true]$val$} and then {\lstinline[mathescape=true]$pair$} that value onto the argument list being accumulated in {\lstinline[mathescape=true]$argl$}. Since we {\lstinline[mathescape=true]$pair$} the arguments onto {\lstinline[mathescape=true]$argl$} in sequence, we must start with the last argument and end with the first, so that the arguments will appear in order from first to last in the resulting list. Rather than waste an instruction by initializing {\lstinline[mathescape=true]$argl$} to the empty list to set up for this sequence of evaluations, we make the first code sequence construct the initial {\lstinline[mathescape=true]$argl$}. The general form of the argument-list construction is thus as follows: \begin{JavaScript} $\langle \textit{compilation of last operand, targeted to}$ val$\rangle$ assign("argl", list(op("list"), reg("val"))); $\langle \textit{compilation of next operand, targeted to}$ val$\rangle$ assign("argl", list(op("pair"), reg("val"), reg("argl"))); $\ldots$ $\langle \textit{compilation of first operand, targeted to}$ val$\rangle$ assign("argl", list(op("pair"), reg("val"), reg("argl"))); \end{JavaScript}

{\lstinline[mathescape=true]$Argl$} must be preserved around each operand evaluation except the first (so that arguments accumulated so far won't be lost), and {\lstinline[mathescape=true]$env$} must be preserved around each operand evaluation except the last (for use by subsequent operand evaluations).

Compiling this argument code is a bit tricky, because of the special treatment of the first operand to be evaluated and the need to preserve {\lstinline[mathescape=true]$argl$} and {\lstinline[mathescape=true]$env$} in different places. The {\lstinline[mathescape=true]$construct_arglist$} function takes as arguments the code that evaluates the individual operands. If there are no operands at all, it simply emits the instruction \begin{JavaScript} assign(argl, list(constant("null"))); \end{JavaScript}

Otherwise, {\lstinline[mathescape=true]$construct_arglist$} creates code that initializes {\lstinline[mathescape=true]$argl$} with the last argument, and appends code that evaluates the rest of the arguments and adjoins them to {\lstinline[mathescape=true]$argl$} in succession. In order to process the arguments from last to first, we must reverse the list of operand code sequences from the order supplied by {\lstinline[mathescape=true]$compile_application$}. \begin{JavaScript} function construct_arglist(operand_codes) { const operand_codes = reverse(operand_codes); if (is_null(operand_codes)) { return make_instruction_sequence( null, list("argl"), list(assign("argl", list(constant("null"))))); } else { const code_to_get_last_arg = append_instruction_sequences( head(operand_codes), make_instruction_sequence( list("val"), list("argl"), list(assign("argl", list(op("list"), reg("val")))))); return is_null(tail(operand_codes)) ? code_to_get_last_arg : preserving( list("env"), code_to_get_last_arg, code_to_get_rest_args(tail(operand_codes))); } } function code_to_get_rest_args(operand_codes) { const code_for_next_arg = preserving( list("argl"), head(operand_codes), make_instruction_sequence( list("val", "argl"), list("argl"), list(assign("argl", list(op("cons"), reg("val"), reg("argl")))))); return is_null(tail(operand_codes)) ? code_for_next_arg : preserving(list("env"), code_for_next_arg, code_to_get_rest_args(tail(operand_codes))); } \end{JavaScript}

\subsubsection{ Applying functions

After evaluating the elements of a combination, the compiled code must apply the function in {\lstinline[mathescape=true]$proc$} to the arguments in {\lstinline[mathescape=true]$argl$}. The code performs essentially the same dispatch as the {\lstinline[mathescape=true]$apply$} function in the meta-circular evaluator of section~\ref{sec:core-of-evaluator} or the {\lstinline[mathescape=true]$apply-dispatch$} entry point in the explicit-control evaluator of section~\ref{sec:procedure-application}. It checks whether the function to be applied is a primitive function or a compiled function. For a primitive function, it uses {\lstinline[mathescape=true]$apply_primitive_function$}; we will see shortly how it handles compiled functions. The function-application code has the following form: \begin{JavaScript} test(op("primitive-procedure"), reg("proc")), branch(label("primitive_branch")), "compiled-branch", $\langle \textit{code to apply compiled procedure with given target and appropriate linkage} \rangle$ "primitive-branch", assign($\langle\textit{target}\rangle$, list(op("apply_primitive_procedure"), reg("proc"), reg("argl"))) $\langle\textit{linkage}\rangle$ "after_call", \end{JavaScript}

Observe that the compiled branch must skip around the primitive branch. Therefore, if the linkage for the original function call was {\lstinline[mathescape=true]$next$}, the compound branch must use a linkage that jumps to a label that is inserted after the primitive branch. (This is similar to the linkage used for the true branch in {\lstinline[mathescape=true]$compile_if$}.) \begin{JavaScript} function compile_procedure_call(target, linkage) { const primitive_branch = make_label("primitive-branch"); const compiled_branch = make_label("compiled-branch"); const after_call = make_label("after-call"); const compiled_linkage = linkage === "next" ? after_call : linkage; return append_instruction_sequences( make_instruction_sequence( list("proc"), list(), list(test(op("primitive_procedure"), reg("proc"), branch(label(primitive_branch))))), parallel_instruction_sequences( append_instruction_sequences( compiled_branch, compile_proc_appl(target, compiled_linkage)), append_instruction_sequences( primitive_branch, end_with_linkage( linkage, make_instruction_sequence(list("proc", "argl"), list(target), assign(target, list(op("apply_primitive_procedure"), reg("proc"), reg("argl"))))))), "after_call"); } \end{JavaScript}

The primitive and compound branches, like the true and false branches in {\lstinline[mathescape=true]$compile_if$}, are appended using {\lstinline[mathescape=true]$parallel_instruction_sequences$} rather than the ordinary {\lstinline[mathescape=true]$append_instruction_sequences$}, because they will not be executed sequentially.

\subsubsection{ Applying compiled functions

The code that handles function application is the most subtle part of the compiler, even though the instruction sequences it generates are very short. A compiled function (as constructed by {\lstinline[mathescape=true]$compile_function_definition_expression$}) has an entry point, which is a label that designates where the code for the function starts. The code at this entry point computes a result in {\lstinline[mathescape=true]$val$} and returns by executing the instruction {\lstinline[mathescape=true]$go_to(reg("continue"))$}. Thus, we might expect the code for a compiled-function application (to be generated by {\lstinline[mathescape=true]$compile_proc_appl$}) with a given target and linkage to look like this if the linkage is a label \begin{JavaScript} assign("continue", list(label("proc_return")),) assign("val", list(op("compiled_procedure_entry"), reg("proc"))), go_to(reg("val")), "proc_return", assign($\langle target\rangle$, list(reg("val"))), // included if target is not val go_to(label($\langle linkage\rangle$)), // linkage code \end{JavaScript} or like this if the linkage is {\lstinline[mathescape=true]$return$}. \begin{JavaScript} save("continue"), assign("continue", list(label("proc_return"))), assign("val", list(op("compiled_procedure_entry"), reg("proc"))), go_to(reg("val")), "proc_return", assign($\langle target\rangle$, list(reg("val"))), // included if target is not val restore("continue"), go_to(reg("continue")), // linkage code \end{JavaScript}

This code sets up {\lstinline[mathescape=true]$continue$} so that the function will return to a label {\lstinline[mathescape=true]$proc_return$} and jumps to the function's entry point. The code at {\lstinline[mathescape=true]$proc_return$} transfers the function's result from {\lstinline[mathescape=true]$val$} to the target register (if necessary) and then jumps to the location specified by the linkage. (The linkage is always {\lstinline[mathescape=true]$return$} or a label, because {\lstinline[mathescape=true]$compile_procedure_call$} replaces a {\lstinline[mathescape=true]$next$} linkage for the compound-procedure branch by an {\lstinline[mathescape=true]$after_call$} label.)

In fact, if the target is not {\lstinline[mathescape=true]$val$}, that is exactly the code our compiler will generate.[1]{\lstinline[mathescape=true]$val$} (the only time the compiler specifies a different register is when targeting the evaluation of an operator to {\lstinline[mathescape=true]$proc$}), so the function result is put directly into the target register and there is no need to return to a special location that copies it. Instead, we simplify the code by setting up {\lstinline[mathescape=true]$continue$} so that the function will return directly to the place specified by the caller's linkage: \begin{JavaScript} $\langle \textit{set~up}\rangle$ continue $\langle\textit{for~linkage}\rangle$ assign("val", list(op("compiled_procedure_entry"), reg("proc"))), go_to(reg("val")), \end{JavaScript} Usually, however, the target is {\lstinline[mathescape=true]$val$} (the only time the compiler specifies a different register is when targeting the evaluation of an operator to {\lstinline[mathescape=true]$proc$}), so the function result is put directly into the target register and there is no need to return to a special location that copies it. Instead, we simplify the code by setting up {\lstinline[mathescape=true]$continue$} so that the function will return directly to the place specified by the caller's linkage: \begin{JavaScript} $\langle \textit{set~up}\rangle$ continue $\langle\textit{for~linkage}\rangle$ assign("val", list(op("compiled_procedure_entry"), reg("proc"))), go_to(reg("val")), \end{JavaScript}

If the linkage is a label, we set up {\lstinline[mathescape=true]$continue$} so that the function will return to that label. (That is, the {\lstinline[mathescape=true]$go_to(reg("continue"))$} the function ends with becomes equivalent to the {\lstinline[mathescape=true]$go_to(label($}$\langle linkage\rangle${\lstinline[mathescape=true]$))$} at {\lstinline[mathescape=true]$proc_return$} above.) \begin{JavaScript} assign("continue", list(label($\langle linkage\rangle$))), assign("val", list(op("compiled_procedure_entry"), reg("proc"))), go_to(reg("val")), \end{JavaScript}

If the linkage is {\lstinline[mathescape=true]$return$}, we don't need to set up {\lstinline[mathescape=true]$continue$} at all: It already holds the desired location. (That is, the {\lstinline[mathescape=true]$go_to(reg("continue"))$} the function ends with goes directly to the place where the {\lstinline[mathescape=true]$go_to(reg("continue"))$} at {\lstinline[mathescape=true]$proc_return$} would have gone.) \begin{JavaScript} assign("val", list(op("compiled_procedure_entry"), reg("proc"))), go_to(reg("val")), \end{JavaScript}

With this implementation of the {\lstinline[mathescape=true]$return$} linkage, the compiler generates tail-recursive code. Calling a function as the final step in a function body does a direct transfer, without saving any information on the stack.

Suppose instead that we had handled the case of a function call with a linkage of {\lstinline[mathescape=true]$return$} and a target of {\lstinline[mathescape=true]$val$} as shown above for a non-{\lstinline[mathescape=true]$val$} target. This would destroy tail recursion. Our system would still give the same value for any expression. But each time we called a function, we would save {\lstinline[mathescape=true]$continue$} and return after the call to undo the (useless) save. These extra saves would accumulate during a nest of function calls.[2]

{\lstinline[mathescape=true]$Compile_proc_appl$} generates the above function-application code by considering four cases, depending on whether the target for the call is {\lstinline[mathescape=true]$val$} and whether the linkage is {\lstinline[mathescape=true]$return$}. Observe that the instruction sequences are declared to modify all the registers, since executing the function body can change the registers in arbitrary ways.[3]{\lstinline[mathescape=true]$val$} and linkage {\lstinline[mathescape=true]$return$} is declared to need {\lstinline[mathescape=true]$continue$}: Even though {\lstinline[mathescape=true]$continue$} is not explicitly used in the two-instruction sequence, we must be sure that {\lstinline[mathescape=true]$continue$} will have the correct value when we enter the compiled function. \begin{JavaScript} function compile_proc_appl(target, linkage) { if (target === "val" && linkage !== "return") { return make_instruction_sequence( list("proc"), all_regs, list( assign("continue", list(label(linkage))), assign("val", list(op("compiled_procedure_entry"), reg("proc"))), go_to(reg("val")))); } else if (target !== "val" && linkage !== "return") { const proc_return = make_label("proc_return"); return make_instruction_sequence( list("proc"), all_regs, list( assign("continue", list(label(proc_return))), assign("val", list(op("compiled_procedure_entry"), reg("proc"))), go_to(reg("val")), proc_return, assign(target, list(reg("val"))), go_to(label(linkage)))); } else if (target === "val" && linkage === "return") { return make_instruction_sequence( list("proc", "continue"), all_regs, list( assign("val", list(op("compiled_procedure_entry"), reg("proc"))), go_to(reg("val")))); } else if (target !== "val" && linkage === "return") { error(target, "return linkage, target not val - - COMPILE"); } } \end{JavaScript} Also note that the code sequence for the case with target {\lstinline[mathescape=true]$val$} and linkage {\lstinline[mathescape=true]$return$} is declared to need {\lstinline[mathescape=true]$continue$}: Even though {\lstinline[mathescape=true]$continue$} is not explicitly used in the two-instruction sequence, we must be sure that {\lstinline[mathescape=true]$continue$} will have the correct value when we enter the compiled function. \begin{JavaScript} function compile_proc_appl(target, linkage) { if (target === "val" && linkage !== "return") { return make_instruction_sequence( list("proc"), all_regs, list( assign("continue", list(label(linkage))), assign("val", list(op("compiled_procedure_entry"), reg("proc"))), go_to(reg("val")))); } else if (target !== "val" && linkage !== "return") { const proc_return = make_label("proc_return"); return make_instruction_sequence( list("proc"), all_regs, list( assign("continue", list(label(proc_return))), assign("val", list(op("compiled_procedure_entry"), reg("proc"))), go_to(reg("val")), proc_return, assign(target, list(reg("val"))), go_to(label(linkage)))); } else if (target === "val" && linkage === "return") { return make_instruction_sequence( list("proc", "continue"), all_regs, list( assign("val", list(op("compiled_procedure_entry"), reg("proc"))), go_to(reg("val")))); } else if (target !== "val" && linkage === "return") { error(target, "return linkage, target not val - - COMPILE"); } } \end{JavaScript}

[3] Actually, we signal an error when the target is not {\lstinline[mathescape=true]$val$} and the linkage is {\lstinline[mathescape=true]$return$}, since the only place we request {\lstinline[mathescape=true]$return$} linkages is in compiling functions, and our convention is that functions return their values in {\lstinline[mathescape=true]$val$}.
[3] Making a compiler generate tail-recursive code might seem like a straightforward idea. But most compilers for common languages, including C and Pascal, do not do this, and therefore these languages cannot represent iterative processes in terms of function call alone. The difficulty with tail recursion in these languages is that their implementations use the stack to store function arguments and local variables as well as return addresses. The Scheme implementations described in this book store arguments and variables in memory to be garbage-collected. The reason for using the stack for variables and arguments is that it avoids the need for garbage collection in languages that would not otherwise require it, and is generally believed to be more efficient. Sophisticated JavaScript compilers can, in fact, use the stack for arguments without destroying tail recursion. (See Hanson 1990 for a description.) There is also some debate about whether stack allocation is actually more efficient than garbage collection in the first place, but the details seem to hinge on fine points of computer architecture. (See Appel 1987 and Miller and Rozas 1994 for opposing views on this issue.)
