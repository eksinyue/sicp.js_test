4.4.4 Implementing the Query System

Section~\ref{sec:how-query-works} described how the query system works. Now we fill in the details by presenting a complete implementation of the system.

\subsubsection{ 4.4.4.1~~The Driver Loop and Instantiation

The driver loop for the query system repeatedly reads input expressions. If the expression is a rule or assertion to be added to the data base, then the information is added. Otherwise the expression is assumed to be a query. The driver passes this query to the evaluator {\lstinline[mathescape=true]$qeval$} together with an initial frame stream consisting of a single empty frame. The result of the evaluation is a stream of frames generated by satisfying the query with variable values found in the data base. These frames are used to form a new stream consisting of copies of the original query in which the variables are instantiated with values supplied by the stream of frames, and this final stream is printed at the terminal: \begin{JavaScript} const input_prompt = "// Query input:"; const output_prompt = "// Query results:"; function query_driver_loop() { const input = prompt(input_prompt); const q = query_syntax_process(parse(input)); if (assertion_to_be_added(q)) { add_rule_or_assertion(add_assertion_body(q)); display("Assertion added to data base."); } else { display(output_prompt); display_stream( stream_map( frame => instantiate(q, frame, (v, f) => contract_question_mark(v)), qeval(q, singleton_stream(null)))); } query_driver_loop(); } \end{JavaScript}

Here, as in the other evaluators in this chapter, we use an abstract syntax for the expressions of the query language. The implementation of the expression syntax, including the predicate {\lstinline[mathescape=true]$assertion_to_be_added$} and the selector {\lstinline[mathescape=true]$add_assertion_body$}, is given in section~4.4.4.7. The function {\lstinline[mathescape=true]$add_rule_or_assertion$} is defined in section~4.4.4.5.

Before doing any processing on an input expression, the driver loop transforms it syntactically into a form that makes the processing more efficient. This involves changing the representation of pattern variables. When the query is instantiated, any variables that remain unbound are transformed back to the input representation before being printed. These transformations are performed by the two functions {\lstinline[mathescape=true]$query_syntax_process$} and {\lstinline[mathescape=true]$contract_question_mark$} (section~4.4.4.7).

To instantiate an expression, we copy it, replacing any variables in the expression by their values in a given frame. The values are themselves instantiated, since they could contain variables (for example, if {\lstinline[mathescape=true]$x$} in {\lstinline[mathescape=true]$exp$} is bound to {\lstinline[mathescape=true]$y$} as the result of unification and {\lstinline[mathescape=true]$y$} is in turn bound to~5). The action to take if a variable cannot be instantiated is given by a procedural argument to {\lstinline[mathescape=true]$instantiate$}. \begin{JavaScript} function instantiate(exp, frame, unbound_var_handler) { function copy(exp) { if (is_var(exp)) { const binding = binding_in_frame(exp, frame); if (! binding === undefined) { return unbound_var_handler(exp, frame); } else { return copy(binding_value(binding)); } } else if (is_pair(exp)) { return pair(copy(head(exp)), copy(tail(exp))); } else { return exp; } } copy(exp); } \end{JavaScript}

The functions that manipulate bindings are defined in section~4.4.4.8.

\subsubsection{ 4.4.4.2~~The Evaluator

The {\lstinline[mathescape=true]$qeval$} function, called by the {\lstinline[mathescape=true]$query_driver_loop$}, is the basic evaluator of the query system. It takes as inputs a query and a stream of frames, and it returns a stream of extended frames. It identifies special forms by a data-directed dispatch using {\lstinline[mathescape=true]$get$} and {\lstinline[mathescape=true]$put$}, just as we did in implementing generic operations in chapter~2. Any query that is not identified as a special form is assumed to be a simple query, to be processed by {\lstinline[mathescape=true]$simple_query$}. \begin{JavaScript} function qeval(query, frame_stream) { const qfun = get(type(query), "qeval"); return qfun === undefined ? simple_query(query, frame_stream) : qfun(contents(query), frame_stream); } \end{JavaScript} The functions {\lstinline[mathescape=true]$type$} and {\lstinline[mathescape=true]$contents$}, defined in section~4.4.4.7, implement the abstract syntax of the expressions.

\subsubsection{ Simple queries

The {\lstinline[mathescape=true]$simple_query$} function handles simple queries. It takes as arguments a simple query (a pattern) together with a stream of frames, and it returns the stream formed by extending each frame by all data-base matches of the query. \begin{JavaScript} function simple_query(query_pattern, frame_stream) { return stream_flatmap( frame => stream_append_delayed(find_assertions(query_pattern, frame), delay(apply_rules(query_pattern, frame))), frame_stream); } \end{JavaScript}

For each frame in the input stream, we use {\lstinline[mathescape=true]$find_assertions$} (section~4.4.4.3) to match the pattern against all assertions in the data base, producing a stream of extended frames, and we use {\lstinline[mathescape=true]$apply_rules$} (section~4.4.4.4) to apply all possible rules, producing another stream of extended frames. These two streams are combined (using {\lstinline[mathescape=true]$stream_append_delayed$}, section~4.4.4.6) to make a stream of all the ways that the given pattern can be satisfied consistent with the original frame (see exercise~\ref{ex:q-why-not-append}). The streams for the individual input frames are combined using {\lstinline[mathescape=true]$stream_flatmap$} (section~4.4.4.6) to form one large stream of all the ways that any of the frames in the original input stream can be extended to produce a match with the given pattern.

\subsubsection{ Compound queries

{\lstinline[mathescape=true]$And$} queries are handled as illustrated in Figure~\ref{fig:query-and} by the {\lstinline[mathescape=true]$conjoin$} function, which takes as inputs the conjuncts and the frame stream and returns the stream of extended frames. First, {\lstinline[mathescape=true]$conjoin$} processes the stream of frames to find the stream of all possible frame extensions that satisfy the first query in the conjunction. Then, using this as the new frame stream, it recursively applies {\lstinline[mathescape=true]$conjoin$} to the rest of the queries. \begin{JavaScript} function conjoin(conjuncts, frame_stream) { return is_empty_conjunction(conjuncts) ? frame_stream : conjoin(rest_conjuncts(conjuncts), qeval(first_conjunct(conjuncts), frame_stream)); } \end{JavaScript}

The expression \begin{JavaScript} put("and", "qeval", conjoin); \end{JavaScript} sets up {\lstinline[mathescape=true]$qeval$} to dispatch to {\lstinline[mathescape=true]$conjoin$} when an {\lstinline[mathescape=true]$and$} expression is encountered.

{\lstinline[mathescape=true]$Or$} queries are handled similarly, as shown in Figure~\ref{fig:query-or}. The output streams for the various disjuncts of the {\lstinline[mathescape=true]$or$} are computed separately and merged using the {\lstinline[mathescape=true]$interleave_delayed$} function from section~4.4.4.6. (See exercises~\ref{ex:q-why-not-append} and~\ref{ex:q-why-interleave}.) \begin{JavaScript} function disjoin(disjuncts, frame_stream) { return is_empty_disjunction(disjuncts) ? null : interleave_delayed( qeval(first_disjunct(disjuncts), frame_stream), delay(disjoin(rest_disjuncts(disjuncts), frame_stream))); } put("or", "qeval", disjoin); \end{JavaScript}

The predicates and selectors for the syntax of conjuncts and disjuncts are given in section~4.4.4.7.

\subsubsection{ Filters

{\lstinline[mathescape=true]$Not$} is handled by the method outlined in section~\ref{sec:how-query-works}. We attempt to extend each frame in the input stream to satisfy the query being negated, and we include a given frame in the output stream only if it cannot be extended. \begin{JavaScript} function negate(operands, frame_stream) { return stream_flatmap( frame => is_null(qeval(negated_query(operands), singleton_stream(frame))) ? singleton_stream(frame) : null, frame_stream); } put("not", "qeval", negate); \end{JavaScript}

{\lstinline[mathescape=true]$Javascript_value$} is a filter similar to {\lstinline[mathescape=true]$not$}. Each frame in the stream is used to instantiate the variables in the pattern, the indicated predicate is applied, and the frames for which the predicate returns false are filtered out of the input stream. An error results if there are unbound pattern variables. \begin{JavaScript} function javascript_value(call, frame_stream) { return stream_flatmap( frame => execute(instantiate(call, frame, (v, f) => error(v, "Unknown pat var --- javascript_value"))) ? singleton_stream(frame) : null, frame_stream); } put("javascript_value", "qeval", javascript_value); \end{JavaScript}

The function {\lstinline[mathescape=true]$execute$}, which applies the predicate to the arguments, must {\lstinline[mathescape=true]$evaluate$} the predicate expression to get the function to apply. However, it must not evaluate the arguments, since they are already the actual arguments, not expressions whose evaluation (in JavaScript) will produce the arguments. Note that {\lstinline[mathescape=true]$execute$} is implemented using {\lstinline[mathescape=true]$evaluate$} and {\lstinline[mathescape=true]$apply$} from the underlying JavaScript system. \begin{JavaScript} function execute(exp) { return apply(evaluate(predicate(exp), user_initial_environment), args(exp)); } \end{JavaScript}

The {\lstinline[mathescape=true]$always_true$} expression provides for a query that is always satisfied. It ignores its contents (normally empty) and simply passes through all the frames in the input stream. The {\lstinline[mathescape=true]$always_true$} expression is used by the {\lstinline[mathescape=true]$rule_body$} selector (section~4.4.4.7) to provide bodies for rules that were defined without bodies (that is, rules whose bodies are always satisfied). \begin{JavaScript} function always_true(ignore, frame_stream) { return frame_stream; } put("always_true", "qeval", always_true); \end{JavaScript}

The selectors that define the syntax of {\lstinline[mathescape=true]$not$} and {\lstinline[mathescape=true]$javascript_value$} are given in section~4.4.4.7.

\subsubsection{ 4.4.4.3~~Finding Assertions by Pattern Matching

The function {\lstinline[mathescape=true]$find_assertions$}, called by {\lstinline[mathescape=true]$simple_query$} (section~4.4.4.2), takes as input a pattern and a frame. It returns a stream of frames, each extending the given one by a data-base match of the given pattern. It uses {\lstinline[mathescape=true]$fetch_assertions$} (section~4.4.4.5) to get a stream of all the assertions in the data base that should be checked for a match against the pattern and the frame. The reason for {\lstinline[mathescape=true]$fetch_assertions$} here is that we can often apply simple tests that will eliminate many of the entries in the data base from the pool of candidates for a successful match. The system would still work if we eliminated {\lstinline[mathescape=true]$fetch_assertions$} and simply checked a stream of all assertions in the data base, but the computation would be less efficient because we would need to make many more calls to the matcher. \begin{JavaScript} function find_assertions(pattern, frame) { return stream_flatmap( datum => check_an_assertion(datum, pattern, frame), fetch_assertions(pattern, frame)); } \end{JavaScript}

The function {\lstinline[mathescape=true]$check_an_assertion$} takes as arguments a pattern, a data object (assertion), and a frame and returns either a one-element stream containing the extended frame or {\lstinline[mathescape=true]$null$} if the match fails. \begin{JavaScript} function check_an_assertion(assertion, query_pat, query_frame) { const match_result = pattern_match(query_pat, assertion, query_frame); return match_result === "failed" ? null : singleton_stream(match_result); } \end{JavaScript}

The basic pattern matcher returns either the string {\lstinline[mathescape=true]$"failed"$} or an extension of the given frame. The basic idea of the matcher is to check the pattern against the data, element by element, accumulating bindings for the pattern variables. If the pattern and the data object are the same, the match succeeds and we return the frame of bindings accumulated so far. Otherwise, if the pattern is a variable we extend the current frame by binding the variable to the data, so long as this is consistent with the bindings already in the frame. If the pattern and the data are both pairs, we (recursively) match the {\lstinline[mathescape=true]$head$} of the pattern against the {\lstinline[mathescape=true]$head$} of the data to produce a frame; in this frame we then match the {\lstinline[mathescape=true]$tail$} of the pattern against the {\lstinline[mathescape=true]$tail$} of the data. If none of these cases are applicable, the match fails and we return the symbol {\lstinline[mathescape=true]$failed$}. \begin{JavaScript} function pattern_match(pat, dat, frame) { return frame === "failed" ? "failed" : equal(pat, dat) ? frame : is_var(pat) ? extend_if_consistent(pat, dat, frame) : is_pair(pat) && is_pair(dat) ? pattern_match(tail(pat), tail(dat), pattern_match(head(pat), head(dat), frame)) : "failed"; } \end{JavaScript}

Here is the function that extends a frame by adding a new binding, if this is consistent with the bindings already in the frame: \begin{JavaScript} function extend_if_consistent(variable, dat, frame) { const binding = binding_in_frame(variable, frame); return binding !== undefined ? extend(variable, dat, frame) : pattern_match(binding_value(binding), dat, frame); } \end{JavaScript}

If there is no binding for the variable in the frame, we simply add the binding of the variable to the data. Otherwise we match, in the frame, the data against the value of the variable in the frame. If the stored value contains only constants, as it must if it was stored during pattern matching by {\lstinline[mathescape=true]$extend_if_consistent$}, then the match simply tests whether the stored and new values are the same. If so, it returns the unmodified frame; if not, it returns a failure indication. The stored value may, however, contain pattern variables if it was stored during unification (see section~4.4.4.4). The recursive match of the stored pattern against the new data will add or check bindings for the variables in this pattern. For example, suppose we have a frame in which {\lstinline[mathescape=true]$x$} is bound to {\lstinline[mathescape=true]$list("f", y)$} and {\lstinline[mathescape=true]$y$} is unbound, and we wish to augment this frame by a binding of {\lstinline[mathescape=true]$x$} to {\lstinline[mathescape=true]$list("f", "b")$}. We look up {\lstinline[mathescape=true]$x$} and find that it is bound to {\lstinline[mathescape=true]$list("f", y)$}. This leads us to match {\lstinline[mathescape=true]$list("f", y)$} against the proposed new value {\lstinline[mathescape=true]$list("f", "b")$} in the same frame. Eventually this match extends the frame by adding a binding of {\lstinline[mathescape=true]$y$} to {\lstinline[mathescape=true]$"b"$}. The variable {\lstinline[mathescape=true]$x$} remains bound to {\lstinline[mathescape=true]$list("f", y)$}. We never modify a stored binding and we never store more than one binding for a given variable.

The functions used by {\lstinline[mathescape=true]$extend_if_consistent$} to manipulate bindings are defined in section~4.4.4.8.

\subsubsection{ 4.4.4.4~~Rules and Unification

The function {\lstinline[mathescape=true]$apply_rules$} is the rule analog of {\lstinline[mathescape=true]$find_assertions$} (section~4.4.4.3). It takes as input a pattern and a frame, and it forms a stream of extension frames by applying rules from the data base. The function {\lstinline[mathescape=true]$stream_flatmap$} maps {\lstinline[mathescape=true]$apply_a_rule$} down the stream of possibly applicable rules (selected by {\lstinline[mathescape=true]$fetch_rules$}, section~4.4.4.5) and combines the resulting streams of frames. \begin{JavaScript} function apply_rules(pattern, frame) { return stream_flatmap( rule => apply_a_rule(rule, pattern, frame), fetch_rules(pattern, frame)); } \end{JavaScript}

The function {\lstinline[mathescape=true]$apply_a_rule$} applies rules using the method outlined in section \ref{sec:how-query-works}. It first augments its argument frame by unifying the rule conclusion with the pattern in the given frame. If this succeeds, it evaluates the rule body in this new frame.

Before any of this happens, however, the program renames all the variables in the rule with unique new names. The reason for this is to prevent the variables for different rule applications from becoming confused with each other. For instance, if two rules both use a variable named {\lstinline[mathescape=true]$x$}, then each one may add a binding for {\lstinline[mathescape=true]$x$} to the frame when it is applied. These two {\lstinline[mathescape=true]$x$}'s have nothing to do with each other, and we should not be fooled into thinking that the two bindings must be consistent. Rather than rename variables, we could devise a more clever environment structure; however, the renaming approach we have chosen here is the most straightforward, even if not the most efficient. (See exercise~\ref{ex:query-local-names}.) Here is the {\lstinline[mathescape=true]$apply_a_rule$} function: \begin{JavaScript} function apply_a_rule(rule, query_pattern, query_frame) { const clean_rule = rename_variables_in(rule); const unify_result = unify_match(query_pattern, conclusion(clean_rule), query_frame); return unify_result === "failed" ? null : qeval(rule_body(clean_rule), singleton_stream(unify_result)); } \end{JavaScript}

The selectors {\lstinline[mathescape=true]$rule_body$} and {\lstinline[mathescape=true]$conclusion$} that extract parts of a rule are defined in section~4.4.4.7.

We generate unique variable names by associating a unique identifier (such as a number) with each rule application and combining this identifier with the original variable names. For example, if the rule-application identifier is 7, we might change each {\lstinline[mathescape=true]$x$} in the rule to {\lstinline[mathescape=true]$x_7$} and each {\lstinline[mathescape=true]$y$} in the rule to {\lstinline[mathescape=true]$y_7$}. (The functions {\lstinline[mathescape=true]$make_new_variable$} and {\lstinline[mathescape=true]$new_rule_application_id$} are included with the syntax functions in section~4.4.4.7.) \begin{JavaScript} function rename_variables_in(rule) { const rule_application_id = new_rule_application_id(); function tree_walk(exp) { return is_var(exp) ? make_new_variable(exp, rule_application_id) : is_pair(exp) ? pair(tree_walk(head(exp)), tree_walk(tail(exp))) : exp; } tree_walk(rule); } \end{JavaScript}

The unification algorithm is implemented as a function that takes as inputs two patterns and a frame and returns either the extended frame or the string {\lstinline[mathescape=true]$"failed"$}. The unifier is like the pattern matcher except that it is symmetricalâ€”variables are allowed on both sides of the match. The function {\lstinline[mathescape=true]$unify_match$} is basically the same as {\lstinline[mathescape=true]$pattern_match$}, except that there is extra code (marked {\lstinline[mathescape=true]$***$} below) to handle the case where the object on the right side of the match is a variable. \begin{JavaScript} function unify_match(p1, p2, frame) { return frame === "failed" ? "failed" : equal(p1, p2) ? frame : is_var(p1) ? extend_if_possible(p1, p2, frame) : is_var(p2) ? extend_if_possible(p2, p1, frame) // *** : pair(p1) && pair(p2) ? unify_match(tail(p1), tail(p2), unify_match(head(p1), head(p2), frame)) : "failed"; } \end{JavaScript}

In unification, as in one-sided pattern matching, we want to accept a proposed extension of the frame only if it is consistent with existing bindings. The function {\lstinline[mathescape=true]$extend_if_possible$} used in unification is the same as the {\lstinline[mathescape=true]$extend_if_consistent$} used in pattern matching except for two special checks, marked {\lstinline[mathescape=true]$***$} in the program below. In the first case, if the variable we are trying to match is not bound, but the value we are trying to match it with is itself a (different) variable, it is necessary to check to see if the value is bound, and if so, to match its value. If both parties to the match are unbound, we may bind either to the other.

The second check deals with attempts to bind a variable to a pattern that includes that variable. Such a situation can occur whenever a variable is repeated in both patterns. Consider, for example, unifying the two patterns {\lstinline[mathescape=true]$list(x, x)$} and {\lstinline[mathescape=true]$list(y,$} $\textit{expression involving}$ {\lstinline[mathescape=true]$y)$} in a frame where both {\lstinline[mathescape=true]$x$} and {\lstinline[mathescape=true]$y$} are unbound. First {\lstinline[mathescape=true]$x$} is matched against {\lstinline[mathescape=true]$y$}, making a binding of {\lstinline[mathescape=true]$x$} to {\lstinline[mathescape=true]$y$}. Next, the same {\lstinline[mathescape=true]$x$} is matched against the given expression involving {\lstinline[mathescape=true]$y$}. Since {\lstinline[mathescape=true]$x$} is already bound to {\lstinline[mathescape=true]$y$}, this results in matching {\lstinline[mathescape=true]$y$} against the expression. If we think of the unifier as finding a set of values for the pattern variables that make the patterns the same, then these patterns imply instructions to find a {\lstinline[mathescape=true]$y$} such that {\lstinline[mathescape=true]$y$} is equal to the $\textit{expression involving}${\lstinline[mathescape=true]$y$}. There is no general method for solving such equations, so we reject such bindings; these cases are recognized by the predicate {\lstinline[mathescape=true]$depends_on$}.[1]{\lstinline[mathescape=true]$list(x, x)$} and {\lstinline[mathescape=true]$list(y, y)$}. The second attempt to bind {\lstinline[mathescape=true]$x$} to {\lstinline[mathescape=true]$y$} matches {\lstinline[mathescape=true]$y$} (the stored value of {\lstinline[mathescape=true]$x$}) against {\lstinline[mathescape=true]$y$} (the new value of {\lstinline[mathescape=true]$x$}). This is taken care of by the {\lstinline[mathescape=true]$equal$} clause of {\lstinline[mathescape=true]$unify_match$}. \begin{JavaScript} function extend_if_possible(variable, val, frame) { const binding = binding_in_frame(variable, frame); if (binding !== undefined) { return unify_match(binding_value(binding), val, frame); } else if (is_var(val)) { // *** const binding = binding_in_frame(val, frame); return binding !== undefined ? unify_match(variable, binding_value(binding), frame) : extend(variable, val, frame); } else if (depends_on(val, variable, frame)) { // *** return "failed"; } else { return extend(variable, val, frame); } } \end{JavaScript} On the other hand, we do not want to reject attempts to bind a variable to itself. For example, consider unifying {\lstinline[mathescape=true]$list(x, x)$} and {\lstinline[mathescape=true]$list(y, y)$}. The second attempt to bind {\lstinline[mathescape=true]$x$} to {\lstinline[mathescape=true]$y$} matches {\lstinline[mathescape=true]$y$} (the stored value of {\lstinline[mathescape=true]$x$}) against {\lstinline[mathescape=true]$y$} (the new value of {\lstinline[mathescape=true]$x$}). This is taken care of by the {\lstinline[mathescape=true]$equal$} clause of {\lstinline[mathescape=true]$unify_match$}. \begin{JavaScript} function extend_if_possible(variable, val, frame) { const binding = binding_in_frame(variable, frame); if (binding !== undefined) { return unify_match(binding_value(binding), val, frame); } else if (is_var(val)) { // *** const binding = binding_in_frame(val, frame); return binding !== undefined ? unify_match(variable, binding_value(binding), frame) : extend(variable, val, frame); } else if (depends_on(val, variable, frame)) { // *** return "failed"; } else { return extend(variable, val, frame); } } \end{JavaScript}

The function {\lstinline[mathescape=true]$depends_on$} is a predicate that tests whether an expression proposed to be the value of a pattern variable depends on the variable. This must be done relative to the current frame because the expression may contain occurrences of a variable that already has a value that depends on our test variable. The structure of {\lstinline[mathescape=true]$depends_on$} is a simple recursive tree walk in which we substitute for the values of variables whenever necessary. \begin{JavaScript} function depends_on(exp, variable, frame) { function tree_walk(e) { if (is_var(e)) { if (equal(variable, e)) { return true; } else { const b = binding_in_frame(e, frame); if (b !== undefined) { return tree_walk(binding_value(b)); } else { return false; } } } else if (is_pair(e)) { return tree_walk(head(e)) || tree_walk(tail(e)); } else { return false; } } return tree_walk(exp); } \end{JavaScript}

\subsubsection{ 4.4.4.5~~Maintaining the Data Base

One important problem in designing logic programming languages is that of arranging things so that as few irrelevant data-base entries as possible will be examined in checking a given pattern. In our system, in addition to storing all assertions in one big stream, we store all assertions whose {\lstinline[mathescape=true]$head$}s are constant symbols in separate streams, in a table indexed by the symbol. To fetch an assertion that may match a pattern, we first check to see if the {\lstinline[mathescape=true]$head$} of the pattern is a constant symbol. If so, we return (to be tested using the matcher) all the stored assertions that have the same {\lstinline[mathescape=true]$head$}. If the pattern's {\lstinline[mathescape=true]$head$} is not a constant symbol, we return all the stored assertions. Cleverer methods could also take advantage of information in the frame, or try also to optimize the case where the {\lstinline[mathescape=true]$head$} of the pattern is not a constant symbol. We avoid building our criteria for indexing (using the {\lstinline[mathescape=true]$head$}, handling only the case of constant symbols) into the program; instead we call on predicates and selectors that embody our criteria. \begin{JavaScript} let THE_ASSERTIONS = null; function fetch_assertions(pattern, frame) { return use_index(pattern) ? get_indexed_assertions(pattern) : get_all_assertions; } function get_all_assertions() { return THE_ASSERTIONS; } function get_indexed_assertions(pattern) { return get_stream(index_key_of(pattern), "assertion-stream"); } \end{JavaScript}

The function {\lstinline[mathescape=true]$get_stream$} looks up a stream in the table and returns an empty stream if nothing is stored there. \begin{JavaScript} function get_stream(key1, key2) { const s = get(key1, key2); return s !== undefined ? s : null; } \end{JavaScript}

Rules are stored similarly, using the {\lstinline[mathescape=true]$head$} of the rule conclusion. Rule conclusions are arbitrary patterns, however, so they differ from assertions in that they can contain variables. A pattern whose {\lstinline[mathescape=true]$head$} is a constant symbol can match rules whose conclusions start with a variable as well as rules whose conclusions have the same {\lstinline[mathescape=true]$head$}. Thus, when fetching rules that might match a pattern whose {\lstinline[mathescape=true]$head$} is a constant symbol we fetch all rules whose conclusions start with a variable as well as those whose conclusions have the same {\lstinline[mathescape=true]$head$} as the pattern. For this purpose we store all rules whose conclusions start with a variable in a separate stream in our table, indexed by the string {\lstinline[mathescape=true]$"variables"$}. \begin{JavaScript} let THE_RULES = null; function fetch_rules(pattern, frame) { return use_index(pattern) ? get_indexed_rules(pattern) : get_all_rules(); } function get_all_rules() { return THE_RULES; } function get_indexed_rules(pattern) { return stream_append( get_stream(index_key_of(pattern), "rule-stream"), get_stream("variables", "rule-stream")); } \end{JavaScript}

The function {\lstinline[mathescape=true]$add_rule_or_assertion$} is used by {\lstinline[mathescape=true]$query_driver_loop$} to add assertions and rules to the data base. Each item is stored in the index, if appropriate, and in a stream of all assertions or rules in the data base. \begin{JavaScript} function add_rule_or_assertion(assertion) { return is_rule(assertion) ? add_rule(assertion) : add_assertion(assertion); } function add_assertion(assertion) { store_assertion_in_index(assertion); const old_assertions = THE_ASSERTIONS; THE_ASSERTIONS = pair(assertion, () => old_assertions); return "ok"; } function add_rule(rule) { store_rule_in_index(rule); const old_rules = THE_RULES; THE_RULES = pair(rule, () => old_rules); return "ok"; } \end{JavaScript}

To actually store an assertion or a rule, we check to see if it can be indexed.
