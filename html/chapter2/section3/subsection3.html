2.3.3 Example: Representing Sets

In the previous examples we built representations for two kinds of compound data objects: rational numbers and algebraic expressions. In one of these examples we had the choice of simplifying (reducing) the expressions at either construction time or selection time, but other than that the choice of a representation for these structures in terms of lists was straightforward. When we turn to the representation of sets, the choice of a representation is not so obvious. Indeed, there are a number of possible representations, and they differ significantly from one another in several ways.

Informally, a set is simply a collection of distinct objects. To give a more precise definition we can employ the method of data abstraction. That is, we define set by specifying the operations that are to be used on sets. These are {\lstinline[mathescape=true]$union_set$}, {\lstinline[mathescape=true]$intersection_set$}, {\lstinline[mathescape=true]$is_element_of_set$}, and {\lstinline[mathescape=true]$adjoin_set$}. The function {\lstinline[mathescape=true]$is_element_of_set$} is a predicate that determines whether a given element is a member of a set. The function {\lstinline[mathescape=true]$adjoin_set$} takes an object and a set as arguments and returns a set that contains the elements of the original set and also the adjoined element. The function {\lstinline[mathescape=true]$union_set$} computes the union of two sets, which is the set containing each element that appears in either argument. The function {\lstinline[mathescape=true]$intersection_set$} computes the intersection of two sets, which is the set containing only elements that appear in both arguments. From the viewpoint of data abstraction, we are free to design any representation that implements these operations in a way consistent with the interpretations given above.[1]

\subsubsection{ Sets as unordered lists

One way to represent a set is as a list of its elements in which no element appears more than once. The empty set is represented by the empty list. In this representation, {\lstinline[mathescape=true]$is_element_of_set$} is similar to the function {\lstinline[mathescape=true]$member$} of section~\ref{sec:strings}. It uses {\lstinline[mathescape=true]$is_equal$} instead of {\lstinline[mathescape=true]$===$} so that the set elements need not be primitive values: \begin{lrbox}{\lstbox} \begin{JavaScriptClickable} function is_element_of_set(x, set) { return ! is_null(set) && ( is_equal(x, head(set)) || is_element_of_set(x, tail(set)) ); } \end{JavaScriptClickable} \end{lrbox} \href{https://sourceacademy.nus.edu.sg/playground#chap=4&prgrm=GYVwdgxgLglg9mABDAzgfQKYBsMFsNhRpzBooZQAUAHgDSLlQCUiA3gFCJcBOFI3SAITJ0YEFiyVGLAGQzEnLkuWURmAI4gAhpLqIAFhi0ATKRSYsAPpYXK7S1Jhz5CxUoxr0oWmJOksmAG52AF8gA}{\usebox\lstbox}

For {\lstinline[mathescape=true]$intersection_set$} we can use a recursive strategy. In designing a representation, one of the issues we should be concerned with is efficiency. Consider the number of steps required by our set operations. Since they all use {\lstinline[mathescape=true]$is_element_of_set$}, the speed of this operation has a major impact on the efficiency of the set implementation as a whole. Now, in order to check whether an object is a member of a set, {\lstinline[mathescape=true]$is_element_of_set$} may have to scan the entire set. (In the worst case, the object turns out not to be in the set.) Hence, if the set has $n$ elements, {\lstinline[mathescape=true]$is_element_of_set$} might take up to $n$ steps. Thus, the number of steps required grows as $\Theta(n)$. The number of steps required by {\lstinline[mathescape=true]$adjoin-set$}, which uses this operation, also grows as $\Theta(n)$. For {\lstinline[mathescape=true]$intersection_set$}, which does an {\lstinline[mathescape=true]$is_element_of_set$} check for each element of {\lstinline[mathescape=true]$set1$}, the number of steps required grows as the product of the sizes of the sets involved, or $\Theta(n^{2})$ for two sets of size $n$. The same will be true of {\lstinline[mathescape=true]$union_set$}.

One way to speed up our set operations is to change the representation so that the set elements are listed in increasing order. To do this, we need some way to compare two objects so that we can say which is bigger. For example, we could compare symbols lexicographically, or we could agree on some method for assigning a unique number to an object and then compare the elements by comparing the corresponding numbers. To keep our discussion simple, we will consider only the case where the set elements are numbers, so that we can compare elements using {\lstinline[mathescape=true]$>$} and {\lstinline[mathescape=true]$<$}. We will represent a set of numbers by listing its elements in increasing order. Whereas our first representation above allowed us to represent the set $\{1,3,6,10\}$ by listing the elements in any order, our new representation allows only the list {\lstinline[mathescape=true]$list(1, 3, 6, 10)$}.

One advantage of ordering shows up in {\lstinline[mathescape=true]$is_element_of_set$}: In checking for the presence of an item, we no longer have to scan the entire set. If we reach a set element that is larger than the item we are looking for, then we know that the item is not in the set: \begin{lrbox}{\lstbox} \begin{JavaScriptClickable} function is_element_of_set(x,set) { return ! is_null(set) && ( x === head(set)) ? true : x < head(set) ? false \end{JavaScriptClickable} \end{lrbox} \begin{JavaScriptClickable} /*!\href{https://sourceacademy.nus.edu.sg/playground#chap=4&prgrm=GYVwdgxgLglg9mABDAzgfQKYBsMFsNhRpzBooZQAUAHgDTlQCUiA3gFCKeIBOFI3SAITJ0YEFiyUGzAGQzEHLkq6VE1RAF4tiABYYAhgBMpFRs0XLLAfkRRuIDBctKAXGsQAeXQePSnzpRtgfSxyfwDON1RMHHxCYlIGGlpbfRhJaUYAbjYAXyA}{\usebox\lstbox}!*/ : is_element_of_set(x, tail(set)); } \end{JavaScriptClickable}

How many steps does this save? In the worst case, the item we are looking for may be the largest one in the set, so the number of steps is the same as for the unordered representation. On the other hand, if we search for items of many different sizes we can expect that sometimes we will be able to stop searching at a point near the beginning of the list and that other times we will still need to examine most of the list. On the average we should expect to have to examine about half of the items in the set. Thus, the average number of steps required will be about $n/2$. This is still $\Theta(n)$ growth, but it does save us, on the average, a factor of 2 in number of steps over the previous implementation.

We obtain a more impressive speedup with {\lstinline[mathescape=true]$intersection_set$}. In the unordered representation this operation required $\Theta(n^2)$ steps, because we performed a complete scan of {\lstinline[mathescape=true]$set2$} for each element of {\lstinline[mathescape=true]$set1$}. But with the ordered representation, we can use a more clever method. Begin by comparing the initial elements, {\lstinline[mathescape=true]$x1$} and {\lstinline[mathescape=true]$x2$}, of the two sets. If {\lstinline[mathescape=true]$x1$} equals {\lstinline[mathescape=true]$x2$}, then that gives an element of the intersection, and the rest of the intersection is the intersection of the {\lstinline[mathescape=true]$tail$}s of the two sets. Suppose, however, that {\lstinline[mathescape=true]$x1$} is less than {\lstinline[mathescape=true]$x2$}. Since {\lstinline[mathescape=true]$x2$} is the smallest element in {\lstinline[mathescape=true]$set2$}, we can immediately conclude that {\lstinline[mathescape=true]$x1$} cannot appear anywhere in {\lstinline[mathescape=true]$set2$} and hence is not in the intersection. Hence, the intersection is equal to the intersection of {\lstinline[mathescape=true]$set2$} with the {\lstinline[mathescape=true]$tail$} of {\lstinline[mathescape=true]$set1$}. Similarly, if {\lstinline[mathescape=true]$x2$} is less than {\lstinline[mathescape=true]$x1$}, then the intersection is given by the intersection of {\lstinline[mathescape=true]$set1$} with the {\lstinline[mathescape=true]$tail$} of {\lstinline[mathescape=true]$set2$}. To estimate the number of steps required by this process, observe that at each step we reduce the intersection problem to computing intersections of smaller sets—removing the first element from {\lstinline[mathescape=true]$set1$} or {\lstinline[mathescape=true]$set2$} or both. Thus, the number of steps required is at most the sum of the sizes of {\lstinline[mathescape=true]$set1$} and {\lstinline[mathescape=true]$set2$}, rather than the product of the sizes as with the unordered representation. This is $\Theta(n)$ growth rather than $\Theta(n^2)$—a considerable speedup, even for sets of moderate size.

We can do better than the ordered-list representation by arranging the set elements in the form of a tree. Each node of the tree holds one element of the set, called the entry at that node, and a link to each of two other (possibly empty) nodes. The left link points to elements smaller than the one at the node, and the right link to elements greater than the one at the node. Figure~\ref{fig:binary-tree} shows some trees that represent the set $\{1,3,5,7,9,11\}$. The same set may be represented by a tree in a number of different ways. The only thing we require for a valid representation is that all elements in the left subtree be smaller than the node entry and that all elements in the right subtree be larger. \begin{figure}[H] \centering \maxsizebox{\linewidth}{0.8\paperheight}{\includegraphics[scale=0.8]{{img\string_original/ch2-Z-G-51}.svg}} \caption{Various binary trees that represent the set $\{ 1,3,5,7,9,11 \}$.} \label{fig:binary-tree} \end{figure}

The advantage of the tree representation is this: Suppose we want to check whether a number $x$ is contained in a set. We begin by comparing $x$ with the entry in the top node. If $x$ is less than this, we know that we need only search the left subtree; if $x$ is greater, we need only search the right subtree. Now, if the tree is balanced, each of these subtrees will be about half the size of the original. Thus, in one step we have reduced the problem of searching a tree of size $n$ to searching a tree of size $n/2$. Since the size of the tree is halved at each step, we should expect that the number of steps needed to search a tree of size $n$ grows as $\Theta(\log n)$.[2] For large sets, this will be a significant speedup over the previous representations.

We can represent trees by using lists. Each node will be a list of three items: the entry at the node, the left subtree, and the right subtree. A left or a right subtree of the empty list will indicate that there is no subtree connected there. We can describe this representation by the following functions:[3] \begin{lrbox}{\lstbox} \begin{JavaScriptClickable} function entry(tree) { return head(tree); } function left_branch(tree) { return head(tail(tree)); } \end{JavaScriptClickable} \end{lrbox} \begin{JavaScriptClickable} /*!\href{https://sourceacademy.nus.edu.sg/playground#chap=4&prgrm=GYVwdgxgLglg9mABAUzFATgTwBQecgSkQG8AoRC9ZKEdJAC2QEMATXKwgblIF9TRIsBIgA2yYFAD6AI3RNI9dviJkKiKjTqJGrXExgilhAtz4Do8JOhgBzelNnyIivIRLlK1Wg2Zso+w38DIwJQ035wC2EAWyYAa2RJV2xUDEwAGjEJdOs7KBUPdS8tERgAZygUtCxM8Sgc23sTXiA}{\usebox\lstbox}!*/ function right_branch(tree) { return head(tail(tail(tree))); } function make_tree(entry,left,right) { return list(entry,left,right); } \end{JavaScriptClickable} \begin{lrbox}{\lstbox} \begin{JavaScriptClickable} function entry(tree) { return head(tree); } function left_branch(tree) { return head(tail(tree)); } \end{JavaScriptClickable} \end{lrbox} \begin{JavaScriptClickable} /*!\href{https://sourceacademy.nus.edu.sg/playground#chap=4&prgrm=GYVwdgxgLglg9mABAUzFATgTwBQecgSkQG8AoRC9ZKEdJAC2QEMATXKwgblIF9TRIsBIgA2yYFAD6AI3RNI9dviJkKiKjTqJGrXExgilhAtz4Do8JOhgBzelNnyIivIRLlK1Wg2Zso+w38DIwJQ035wC2EAWyYAa2RJV2xUDEwAGjEJdOs7KBUPdS8tERgAZygUtCxM8Sgc23sTXiA}{\usebox\lstbox}!*/ function right_branch(tree) { return head(tail(tail(tree))); } function make_tree(entry,left,right) { return list(entry,left,right); } \end{JavaScriptClickable}

Adjoining an item to a set is implemented similarly and also requires $\Theta(\log n)$ steps. To adjoin an item {\lstinline[mathescape=true]$x$}, we compare {\lstinline[mathescape=true]$x$} with the node entry to determine whether {\lstinline[mathescape=true]$x$} should be added to the right or to the left branch, and having adjoined {\lstinline[mathescape=true]$x$} to the appropriate branch we piece this newly constructed branch together with the original entry and the other branch. If {\lstinline[mathescape=true]$x$} is equal to the entry, we just return the node. If we are asked to adjoin {\lstinline[mathescape=true]$x$} to an empty tree, we generate a tree that has {\lstinline[mathescape=true]$x$} as the entry and empty right and left branches. The above claim that searching the tree can be performed in a logarithmic number of steps rests on the assumption that the tree is balanced, i.e., that the left and the right subtree of every tree have approximately the same number of elements, so that each subtree contains about half the elements of its parent. But how can we be certain that the trees we construct will be balanced? Even if we start with a balanced tree, adding elements with {\lstinline[mathescape=true]$adjoin_set$} may produce an unbalanced result. Since the position of a newly adjoined element depends on how the element compares with the items already in the set, we can expect that if we add elements randomly the tree will tend to be balanced on the average. But this is not a guarantee. For example, if we start with an empty set and adjoin the numbers 1 through 7 in sequence we end up with the highly unbalanced tree shown in Figure~\ref{fig:unbalanced-tree}. In this tree all the left subtrees are empty, so it has no advantage over a simple ordered list. One way to solve this problem is to define an operation that transforms an arbitrary tree into a balanced tree with the same elements. Then we can perform this transformation after every few {\lstinline[mathescape=true]$adjoin_set$} operations to keep our set in balance. There are also other ways to solve this problem, most of which involve designing new data structures for which searching and insertion both can be done in $\Theta(\log n)$ steps.[4] \begin{figure}[H] \centering \maxsizebox{\linewidth}{0.8\paperheight}{\includegraphics[scale=0.8]{{img\string_original/ch2-Z-G-52}.svg}} \caption{Unbalanced tree produced by adjoining 1 through 7 in sequence.} \label{fig:unbalanced-tree} \end{figure} \begin{figure}[H] \centering \maxsizebox{\linewidth}{0.8\paperheight}{\includegraphics[scale=0.8]{{img\string_original/ch2-Z-G-52}.svg}} \caption{Unbalanced tree produced by adjoining 1 through 7 in sequence.} \label{fig:unbalanced-tree} \end{figure}

We have examined options for using lists to represent sets and have seen how the choice of representation for a data object can have a large impact on the performance of the programs that use the data. Another reason for concentrating on sets is that the techniques discussed here appear again and again in applications involving information retrieval.

Consider a data base containing a large number of individual records, such as the personnel files for a company or the transactions in an accounting system. A typical data-management system spends a large amount of time accessing or modifying the data in the records and therefore requires an efficient method for accessing records. This is done by identifying a part of each record to serve as an identifying key. A key can be anything that uniquely identifies the record. For a personnel file, it might be an employee's ID number. For an accounting system, it might be a transaction number. Whatever the key is, when we define the record as a data structure we should include a {\lstinline[mathescape=true]$key$} selector function that retrieves the key associated with a given record.

Now we represent the data base as a set of records. To locate the record with a given key we use a function {\lstinline[mathescape=true]$lookup$}, which takes as arguments a key and a data base and which returns the record that has that key, or false if there is no such record. The function {\lstinline[mathescape=true]$lookup$} is implemented in almost the same way as {\lstinline[mathescape=true]$is_element_of_set$}. For example, if the set of records is implemented as an unordered list, we could use \begin{lrbox}{\lstbox} \begin{JavaScriptClickable} function lookup(given_key, set_of_records) { return ! is_null(set_of_records) && ( is_equal(given_key, key(head(set_of_records))) ? head(set_of_records) : lookup(given_key, tail(set_of_records)) ); \end{JavaScriptClickable} \end{lrbox} \begin{JavaScriptClickable} /*!\href{https://sourceacademy.nus.edu.sg/playground#chap=4&prgrm=GYVwdgxgLglg9mABAGznA1iADgCgOYwBuApmAPrrECeANIgM7FRlzBkBOxEc7AJvQEpEAbwBQiCYk5QQ7JAEJEMemTAhkyHI2asOXHvyEAyI+MnnJOJSuIBHEAENNBEuUq1E7nAAtiD3lpMLGyc3HyCAgJmFjEA-Ii+-oE6IfrhUTGZAFwoaJi4LqQU1HRQDjCa2sF6YYYZmZICANyiAL5AA}{\usebox\lstbox}!*/ } \end{JavaScriptClickable}

Of course, there are better ways to represent large sets than as unordered lists. Information-retrieval systems in which records have to be randomly accessed are typically implemented by a tree-based method, such as the binary-tree representation discussed previously. In designing such a system the methodology of data abstraction can be a great help. The designer can create an initial implementation using a simple, straightforward representation such as unordered lists. This will be unsuitable for the eventual system, but it can be useful in providing a quick and dirty data base with which to test the rest of the system. Later on, the data representation can be modified to be more sophisticated. If the data base is accessed in terms of abstract selectors and constructors, this change in representation will not require any changes to the rest of the system.
