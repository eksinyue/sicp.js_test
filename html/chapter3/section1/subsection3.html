3.1.3 The Costs of Introducing Assignment

As we have seen, the assignment statement enables us to model objects that have local state. However, this advantage comes at a price. Our programming language can no longer be interpreted in terms of the substitution model of function application that we introduced in section~\ref{sec:substitution-model}. Moreover, no simple model with nice mathematical properties can be an adequate framework for dealing with objects and assignment in programming languages.

So long as we do not use assignments, two evaluations of the same function with the same arguments will produce the same result, so that functions can be viewed as computing mathematical functions. Programming without any use of assignments, as we did throughout the first two chapters of this book, is accordingly known as functional programming.

To understand how assignment complicates matters, consider a simplified version of the {\lstinline[mathescape=true]$make_withdraw$} function of section~\ref{sec:local-state-variables} that does not bother to check for an insufficient amount: \begin{lrbox}{\lstbox} \begin{JavaScriptClickable} function make_simplified_withdraw(balance) { return amount => { balance = balance - amount; return balance; }; } \end{JavaScriptClickable} \end{lrbox} \href{https://sourceacademy.nus.edu.sg/playground#chap=4&prgrm=GYVwdgxgLglg9mABAWwIYGsCmB9AzjZABwBsZgZMATbAdxigAtKAnVGgCgCNVjVJMAlIgDeAKEQTEzTFBDMkqZHHBREAXgB8I8ZN16J3Xv3WJDfCJkQBaRIuVgoAbh37X02fNM9zmZ690Avs4BQA}{\usebox\lstbox}

Compare this function with the following {\lstinline[mathescape=true]$make_decrementer$} function, which does not use assignment: \begin{lrbox}{\lstbox} \begin{JavaScriptClickable} function make_decrementer(balance) { return amount => balance - amount; } \end{JavaScriptClickable} \end{lrbox} \href{https://sourceacademy.nus.edu.sg/playground#chap=4&prgrm=GYVwdgxgLglg9mABAWwIYGsCmB9AJpiAJ02UzCk0IAoAjVAG1UkwEpEBvAKER8WKhCEkqZHHBREAXgB8iOo2aIAtIhFjyAbk4BfIA}{\usebox\lstbox}

The function {\lstinline[mathescape=true]$make_decrementer$} returns a function that subtracts its input from a designated amount {\lstinline[mathescape=true]$balance$}, but there is no accumulated effect over successive calls, as with {\lstinline[mathescape=true]$make_simplified_withdraw$}: \begin{lrbox}{\lstbox} \begin{JavaScriptClickable} const d = make_decrementer(25); \end{JavaScriptClickable} \end{lrbox} \href{https://sourceacademy.nus.edu.sg/playground#chap=4&prgrm=MYewdgzgLgBAJjAvDAtgQwNYFMD6cvABOWKWYUWhAFAEwCsAlANxA}{\usebox\lstbox} \begin{lrbox}{\lstbox} \begin{JavaScriptClickable} d(20); // output: 5 \end{JavaScriptClickable} \end{lrbox} \href{https://sourceacademy.nus.edu.sg/playground#chap=4&prgrm=CYCgTADAlA3ABAegXA9gVwC4AdMC44CsQA}{\usebox\lstbox} \begin{lrbox}{\lstbox} \begin{JavaScriptClickable} d(10); // output: 15 \end{JavaScriptClickable} \end{lrbox} \href{https://sourceacademy.nus.edu.sg/playground#chap=4&prgrm=CYCgjADAlA3ABAegXA9gVwC4AdMC45gCsQA}{\usebox\lstbox}

We can use the substitution model to explain how {\lstinline[mathescape=true]$make_decrementer$} works. For instance, let us analyze the evaluation of the expression \begin{lrbox}{\lstbox} \begin{JavaScriptClickable} (make_decrementer(25))(20); \end{JavaScriptClickable} \end{lrbox} \href{https://sourceacademy.nus.edu.sg/playground#chap=4&prgrm=BQWwhg1gpg+gJlAxgJyiKA7ALlZwBMArAJTEEAMxA3EA}{\usebox\lstbox} We first simplify the operator of the combination by substituting $25$ for {\lstinline[mathescape=true]$balance$} in the body of {\lstinline[mathescape=true]$make-decrementer$}. This reduces the expression to \begin{lrbox}{\lstbox} \begin{JavaScriptClickable} (amount => 25 - amount)(20); \end{JavaScriptClickable} \end{lrbox} \href{https://sourceacademy.nus.edu.sg/playground#chap=4&prgrm=BQQwtg9grgdgLgAgLwD4ECYCsCC0DzTwCUw6ADEQNxA}{\usebox\lstbox}

Now we apply the operator by substituting 20 for {\lstinline[mathescape=true]$amount$} in the body of the function definition expression: \begin{lrbox}{\lstbox} \begin{JavaScriptClickable} 25 - 20; \end{JavaScriptClickable} \end{lrbox} \href{https://sourceacademy.nus.edu.sg/playground#chap=4&prgrm=EwVgBAtGwAwNxA}{\usebox\lstbox} The final answer is 5.

Observe, however, what happens if we attempt a similar substitution analysis with {\lstinline[mathescape=true]$make_simplified_withdraw$}: \begin{lrbox}{\lstbox} \begin{JavaScriptClickable} (make_simplified_withdraw(25))(20); \end{JavaScriptClickable} \end{lrbox} \href{https://sourceacademy.nus.edu.sg/playground#chap=4&prgrm=BQWwhg1gpg+gzgSxABwDYIGYKgExgdwQBcALHAJzH2ACYBWASgdoAYGBuIA}{\usebox\lstbox}

We first simplify the operator by substituting 25 for {\lstinline[mathescape=true]$balance$} in the return expression of {\lstinline[mathescape=true]$make_simplified_withdraw$}. This reduces the expression to[1] \begin{JavaScript} (amount => { balance = 25 - amount; return 25; })(20); \end{JavaScript} \begin{JavaScript} (amount => { balance = 25 - amount; return 25; })(20); \end{JavaScript}

Now we apply the function by substituting 20 for {\lstinline[mathescape=true]$amount$} in the body of the function: \begin{JavaScript} balance = 25 - 20; return 25; \end{JavaScript}

If we adhered to the substitution model, we would have to say that the meaning of the function application is to first set {\lstinline[mathescape=true]$balance$} to 5 and then return 25 as the value of the expression. This gets the wrong answer. In order to get the correct answer, we would have to somehow distinguish the first occurrence of {\lstinline[mathescape=true]$balance$} (before the effect of the assignment) from the second occurrence of {\lstinline[mathescape=true]$balance$} (after the effect of the assignment), and the substitution model cannot do this.

The trouble here is that substitution is based ultimately on the notion that the symbols in our language are essentially names for values. But as soon as we introduce assignment and the idea that the value of a variable can change, a variable can no longer be simply a name. Now a variable somehow refers to a place where a value can be stored, and the value stored at this place can change. In section~\ref{sec:environment-model} we will see how environments play this role of place in our computational model.

\subsubsection{ Sameness and change

The issue surfacing here is more profound than the mere breakdown of a particular model of computation. As soon as we introduce change into our computational models, many notions that were previously straightforward become problematical. Consider the concept of two things being the same.

Suppose we call {\lstinline[mathescape=true]$make_decrementer$} twice with the same argument to create two functions: \begin{lrbox}{\lstbox} \begin{JavaScriptClickable} const d1 = make_decrementer(25); const d2 = make_decrementer(25); \end{JavaScriptClickable} \end{lrbox} \href{https://sourceacademy.nus.edu.sg/playground#chap=4&prgrm=MYewdgzgLgBAJgRhgXhgWwIYGsCmB9OHYAJxzRzCh2IAoAmAVgEoBuAKDdEljjpXWz5CJMhSq1GrIA}{\usebox\lstbox}

Are {\lstinline[mathescape=true]$d1$} and {\lstinline[mathescape=true]$d2$} the same? An acceptable answer is yes, because {\lstinline[mathescape=true]$d1$} and {\lstinline[mathescape=true]$d2$} have the same computational behaviorâ€”each is a function that subtracts its input from 25. In fact, {\lstinline[mathescape=true]$d1$} could be substituted for {\lstinline[mathescape=true]$d2$} in any computation without changing the result.

Contrast this with making two calls to {\lstinline[mathescape=true]$make_simplified_withdraw$}: \begin{lrbox}{\lstbox} \begin{JavaScriptClickable} const w1 = make_simplified_withdraw(25); const w2 = make_simplified_withdraw(25); \end{JavaScriptClickable} \end{lrbox} \href{https://sourceacademy.nus.edu.sg/playground#chap=4&prgrm=MYewdgzgLgBA7gRhgXhgWwIYGsCmB9CASzQAcAbQgM0JwBM85CoALWgJwzgAoAmAVgCUAbgBQI0JFhweKdNnxFSFanQZNWHbv2FA}{\usebox\lstbox}

Are {\lstinline[mathescape=true]$w1$} and {\lstinline[mathescape=true]$w2$} the same? Surely not, because calls to {\lstinline[mathescape=true]$w1$} and {\lstinline[mathescape=true]$w2$} have distinct effects, as shown by the following sequence of interactions: \begin{lrbox}{\lstbox} \begin{JavaScriptClickable} w1(20); \end{JavaScriptClickable} \end{lrbox} \href{https://sourceacademy.nus.edu.sg/playground#chap=4&prgrm=O4RgFATADAlA3EA}{\usebox\lstbox} \begin{lrbox}{\lstbox} \begin{JavaScriptClickable} w1(20); \end{JavaScriptClickable} \end{lrbox} \href{https://sourceacademy.nus.edu.sg/playground#chap=4&prgrm=O4RgFATADAlA3EA}{\usebox\lstbox} \begin{lrbox}{\lstbox} \begin{JavaScriptClickable} w2(20); \end{JavaScriptClickable} \end{lrbox} \href{https://sourceacademy.nus.edu.sg/playground#chap=4&prgrm=O4JgFCAMCUDcQ}{\usebox\lstbox}

Even though {\lstinline[mathescape=true]$w1$} and {\lstinline[mathescape=true]$w2$} are equal in the sense that they are both created by evaluating the same expression, {\lstinline[mathescape=true]$make_simplified_withdraw(25)$}, it is not true that {\lstinline[mathescape=true]$w1$} could be substituted for {\lstinline[mathescape=true]$w2$} in any expression without changing the result of evaluating the expression.

A language that supports the concept that equals can be substituted for equals in an expression without changing the value of the expression is said to be referentially transparent. Referential transparency is violated when we include assignment in our computer language. This makes it tricky to determine when we can simplify expressions by substituting equivalent expressions. Consequently, reasoning about programs that use assignment becomes drastically more difficult.

Once we forgo referential transparency, the notion of what it means for computational objects to be the same becomes difficult to capture in a formal way. Indeed, the meaning of same in the real world that our programs model is hardly clear in itself. In general, we can determine that two apparently identical objects are indeed the same one only by modifying one object and then observing whether the other object has changed in the same way. But how can we tell if an object has changed other than by observing the same object twice and seeing whether some property of the object differs from one observation to the next? Thus, we cannot determine change without some a priori notion of sameness, and we cannot determine sameness without observing the effects of change.

As an example of how this issue arises in programming, consider the situation where Peter and Paul have a bank account with \$100 in it. There is a substantial difference between modeling this as \begin{lrbox}{\lstbox} \begin{JavaScriptClickable} const peter_acc = make_account(100); const paul_acc = make_account(100); \end{JavaScriptClickable} \end{lrbox} \href{https://sourceacademy.nus.edu.sg/playground#chap=4&prgrm=MYewdgzgLgBADgUyggTgfQIbGDAvDAWwwGsFNsQBXMKACgEYAGRgSgG4AoUSWODSgDbkc+IqWFUaDZuyA}{\usebox\lstbox} and modeling it as \begin{lrbox}{\lstbox} \begin{JavaScriptClickable} const peter_acc = make_account(100); const paul_acc = peter_acc; \end{JavaScriptClickable} \end{lrbox} \href{https://sourceacademy.nus.edu.sg/playground#chap=4&prgrm=MYewdgzgLgBADgUyggTgfQIbGDAvDAWwwGsFNsQBXMKACgEYAGRgSgG4AoUSWODSgDbkc+RMnRZgbIA}{\usebox\lstbox}

In the first situation, the two bank accounts are distinct. Transactions made by Peter will not affect Paul's account, and vice versa. In the second situation, however, we have defined {\lstinline[mathescape=true]$paul_acc$} to be the same thing as {\lstinline[mathescape=true]$peter_acc$} In effect, Peter and Paul now have a joint bank account, and if Peter makes a withdrawal from {\lstinline[mathescape=true]$peter_acc$} Paul will observe less money in {\lstinline[mathescape=true]$paul_acc$}. These two similar but distinct situations can cause confusion in building computational models. With the shared account, in particular, it can be especially confusing that there is one object (the bank account) that has two different names ({\lstinline[mathescape=true]$peter_acc$} and {\lstinline[mathescape=true]$paul_acc$}); if we are searching for all the places in our program where {\lstinline[mathescape=true]$paul_acc$} can be changed, we must remember to look also at things that change {\lstinline[mathescape=true]$peter_acc$}.[2]

With reference to the above remarks on sameness and change, observe that if Peter and Paul could only examine their bank balances, and could not perform operations that changed the balance, then the issue of whether the two accounts are distinct would be moot. In general, so long as we never modify data objects, we can regard a compound data object to be precisely the totality of its pieces. For example, a rational number is determined by giving its numerator and its denominator. But this view is no longer valid in the presence of change, where a compound data object has an identity that is something different from the pieces of which it is composed. A bank account is still the same bank account even if we change the balance by making a withdrawal; conversely, we could have two different bank accounts with the same state information. This complication is a consequence, not of our programming language, but of our perception of a bank account as an object. We do not, for example, ordinarily regard a rational number as a changeable object with identity, such that we could change the numerator and still have the same rational number.

\subsubsection{ Pitfalls of imperative programming

In contrast to functional programming, programming that makes extensive use of assignment is known as imperative programming. In addition to raising complications about computational models, programs written in imperative style are susceptible to bugs that cannot occur in functional programs. For example, recall the iterative factorial program from section~\ref{sec:recursion-and-iteration}: \begin{lrbox}{\lstbox} \begin{JavaScriptClickable} function factorial(n) { function iter(product,counter) { if (counter > n) { return product; } else { return iter(counter*product, \end{JavaScriptClickable} \end{lrbox} \begin{JavaScriptClickable} /*!\href{https://sourceacademy.nus.edu.sg/playground#chap=4&prgrm=GYVwdgxgLglg9mABMAhtOAnGKA2AKMASkQG8AoRS0SWBRGKAUwzwAcM4ATEaAGgjjgmGYuUrj6wRHgFDmiAHyIipChPUZGUEBiTsuPKAG41EgL6JGOAM6NV6jVp1IGzGYLDCAVPu59TDoGBsp7MANQAjIQmDmamceKa2rr0wngRvFEmZkA}{\usebox\lstbox}!*/ counter+1); } } return iter(1,1); } \end{JavaScriptClickable} Instead of passing arguments in the internal iterative loop, we could adopt a more imperative style by using explicit assignment to update the values of the variables {\lstinline[mathescape=true]$product$} and {\lstinline[mathescape=true]$counter$}: \begin{lrbox}{\lstbox} \begin{JavaScriptClickable} function factorial(n) { let product = 1; let counter = 1; function iter() { if (counter > n) { return product; \end{JavaScriptClickable} \end{lrbox} \begin{JavaScriptClickable} /*!\href{https://sourceacademy.nus.edu.sg/playground#chap=4&prgrm=GYVwdgxgLglg9mABMAhtOAnGKA2AKMASkQG8AoRSnAUykQAcM4ATEaRAXkQEYBuCqrUQQ44KNQyce-SsnDR4SGOIx5i5WbJjBEeEWImIAfIiKkBmzRlogMSRizZQZlgL6JqOAM7Vzlyw6s7Fz6YCqIAFQMTEHOFv7ComGGIUnhANTS8f7WULZKKmoumq4WpbK5+YjKEkVkrkA}{\usebox\lstbox}!*/ } else { product = counter * product; counter = counter + 1; return iter(); } } return iter(); } \end{JavaScriptClickable} This does not change the results produced by the program, but it does introduce a subtle trap. How do we decide the order of the assignments? As it happens, the program is correct as written. But writing the assignments in the opposite order \begin{JavaScript} counter = counter + 1; product = counter * product; \end{JavaScript} would have produced a different, incorrect result. In general, programming with assignment forces us to carefully consider the relative orders of the assignments to make sure that each statement is using the correct version of the variables that have been changed. This issue simply does not arise in functional programs.[3]

The complexity of imperative programs becomes even worse if we consider applications in which several processes execute concurrently. We will return to this in section~\ref{sec:time-is-of-the-essence}. First, however, we will address the issue of providing a computational model for expressions that involve assignment, and explore the uses of objects with local state in designing simulations.
